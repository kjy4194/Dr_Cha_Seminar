## 2. Flywheel 사이클 분석

### Step 1: 필요 발생

**어떤 필요가 있었나요?**

> 해외 협력자(미국·유럽)와 화상 회의를 잡을 때, 한국(서울) 기준으로 잡은 날짜·시각이 상대방 타임존에서 언제인지 매번 검색기나 웹 사이트를 여러 번 열어 확인해야 했다. 특히 미 서부·동부·유럽을 동시에 맞출 때 실수가 나지 않도록 한 화면에서 비교하고 싶었다.

**기존에는 어떻게 해결했나요?**

> 구글 검색("KST to EST" 등) 또는 스마트폰 세계시계 앱을 번갈아 사용했다. 회의 안내 메일을 쓸 때는 문장을 직접 조합해 여러 번 고쳐 썼고, 한 번 정리하는 데 약 10~15분이 걸리기도 했다.

---

### Step 2: 도구 생성

**어떤 프롬프트를 사용했나요?**

```
역할: 프론트엔드 개발자
다음 요구사항으로 단일 HTML 파일로도 동작하는 React 웹 앱을 만들어 주세요.
(Artifacts로 실행 가능하게, 외부 API 호출 없이)
[앱 이름] 글로벌 회의 시간 변환기

[기능]

1. 날짜: yyyy-mm-dd 입력 필드 (기본값: 오늘)

2. 시각: HH:mm 입력 (24시간제)

3. 기준 타임존: 드롭다운 — 최소한 "Asia/Seoul", "UTC", "America/New_York" 포함

4. "변환" 버튼 클릭 시, 선택한 기준 날짜·시각을 해당 타임존의 로컬 시각으로 해석한 뒤,

   아래 타임존 각각에 대해 **같은 순간**의 날짜·시각을 표로 표시:

   - Asia/Seoul

   - UTC

   - America/New_York

   - Europe/Berlin

5. 타임존 이름 옆에 약어 표기도 (예: KST, UTC, ET는 대략 표기)

6. 잘못된 날짜/시각이면 빨간색으로 에러 메시지

7. UI는 깔끔한 카드 레이아웃, 반응형
  
[기술]

- React (함수 컴포넌트)

- 시간대 변환은 브라우저 내장 Intl 또는 luxon/dayjs 중 하나로 일관되게 처리

- 한국어 UI 라벨 사용 (날짜, 시각, 기준 타임존, 변환 결과)

[출력]

실행 가능한 전체 코드 한 덩어리로 제공해 주세요.
```

**수정 요청은 몇 번 했나요?**

> **2회** *(실제 횟수로 수정)*

**주요 수정 내용:**

1. 맨아래에 이메일 복사용 한줄 넣기 텍스트 창을 넣어 주세요.

2. 기준 타임 존은 한국으로만 고정해줘

### Step 3: 문제 해결

**앱으로 실제 문제를 해결했나요?**

> ☑ 예 / ☐ 부분적으로 / ☐ 아니오

**해결한 구체적 사례:**

> 가상의 회의 일정(예: 2026-03-20 14:00 KST)을 넣어 미 서부·동부·영국·독일 시각이 한 번에 맞는지 확인했고, 생성된 한 줄 문구를 복사해 메일 초안에 붙여 넣을 수 있음을 검증했다. 검색·수동 작성 대비 같은 정보를 더 짧은 시간에 정리할 수 있었다.

**기존 방식 대비 절감 시간:**

> 기존 **약 3분** (검색·메모 반복) → 앱 사용 **약 10초** (절감: **약 2분**/건)

## 3. 학습 포인트

### 잘된 점

1. **요구사항을 먼저 문장으로 정리**한 뒤 프롬프트에 넣으니, 화면 구성과 기능 범위가 흔들리지 않았다.
2. **30분 목표**로로 “일단 동작”을 우선했고, 세부 기능는 수정 요청으로 단계적으로 맞췄다.

### 어려웠던 점

1. 타임존·서머타임(DST) 해석이 브라우저·라이브러리마다 미묘하게 달라, **한 번 검증**하는 시간이 필요했다.
2. 처음에는 기능을 너무 넣으려다 시간이 늘어날 뻔해, **기준 타임존을 서울로 고정**하고 변환 대상만 4개로 줄여 마무리했다.

### 다음에 시도해볼 것

1. 참가자가 직접 **체크할 타임존**을 고르는 옵션 추가
2. 같은 앱을 **nano banana pro, replit**이나 다른 도구로도 만들어 보고, 프롬프트만 바꿔 재현되는지 비교해 보기
